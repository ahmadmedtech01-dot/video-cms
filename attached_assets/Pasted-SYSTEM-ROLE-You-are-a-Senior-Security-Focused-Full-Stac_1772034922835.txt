FEATURES TO IMPLEMENT (NON-DRM)
====================================================================

A) SECURE HLS PLAYLIST GATEWAY (SIGNED PLAYLIST URL)
1) Frontend must NEVER store or receive origin .m3u8/.ts/.m4s URLs.
2) Frontend calls backend:
   POST /api/video/session
   body: { videoId }
   auth: Firebase ID token
3) Backend validates:
   - user authenticated
   - user has access to videoId/courseId
   - create sessionId and sessionSecret
4) Backend returns:
   {
     sessionId,
     playlistUrl: "https://<your-domain>/hls/<videoId>/master.m3u8?sid=...&st=...&exp=..."
     expiresAt
   }
Where "st" is signed token (HMAC) and exp is short (60s).

B) PLAYLIST REWRITE (PER-SEGMENT TOKENS)
1) Backend endpoint:
   GET /hls/:videoId/master.m3u8
   - verify sid + signature + exp
   - fetch origin master playlist from private origin (server-side)
   - rewrite all variant playlist URLs to your domain:
     /hls/:videoId/variant.m3u8?sid=...&st=...&exp=...
2) Backend endpoint:
   GET /hls/:videoId/:variant.m3u8
   - verify token
   - fetch origin variant playlist
   - rewrite EVERY segment URL into:
     /seg/:videoId/:segmentName?sid=...&st=...&exp=...&n=<nonce>
   - set exp for segments 10–20 seconds
   - ensure tokens are per-segment and can be single-use (nonce tracked)

C) HLS ENCRYPTION KEY GATEWAY (AES-128 / rotating keys)
(Non-DRM encryption)
1) Origin HLS is encrypted with AES-128 (or SAMPLE-AES if supported).
2) The EXT-X-KEY URI must point to our backend:
   /key/:videoId?sid=...&st=...&exp=...
3) Backend key endpoint:
   - verify token and session
   - enforce rate limits
   - return decryption key bytes
4) Rotate keys periodically (ex: every 1–2 minutes) if your packager supports.
If packager rotation not available now, still support single key securely.

D) SEGMENT PROXY (NO DIRECT ORIGIN ACCESS)
1) Endpoint:
   GET /seg/:videoId/:segmentName
   - verify token, nonce, exp
   - enforce concurrency and rate limits per session
   - fetch segment bytes from private origin server-side
   - stream response with correct content-type
2) Add anti-download friction:
   - Max 2 concurrent segment requests per sid
   - If exceeded, flag abuse and revoke session

E) ABUSE DETECTION + "VIDEO PLAYBACK DENIED" OVERLAY
1) Implement an abuse scoring system per session:
   - too many requests/sec
   - too many concurrent connections
   - repeated playlist fetches
   - token re-use from different IP (optional)
   - excessive key endpoint hits
2) When abuse triggers:
   - backend marks session revoked
   - future requests return 403 with code "PLAYBACK_DENIED"
3) Frontend player must show an overlay (no redesign):
   - Semi-transparent overlay above video area only
   - Text: "Video playback denied due to suspicious activity."
   - Button: "Retry" (calls /api/video/session again)
   - On deny: stop/pause playback immediately.
