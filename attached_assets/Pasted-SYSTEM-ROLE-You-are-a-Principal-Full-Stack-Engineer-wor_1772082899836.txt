A) HLS TRANSCODING PIPELINE (FFmpeg)
1) On upload of an MP4 video:
   - Save temporarily to server.
   - Run FFmpeg to generate HLS output:
     - master.m3u8
     - at least ONE variant (e.g. 720p) playlist index_720.m3u8
     - segments in small size by using short duration:
       - target segment duration: 1s (or 2s max)
     - Use HLS format that works broadly:
       - MPEG-TS segments (.ts) OR fMP4 (.m4s). Pick one and be consistent.
2) Ensure segment size is “KB to a few hundred KB” typical (depends bitrate).
3) Store metadata in DB:
   - videoId
   - original filename
   - duration (if available)
   - hlsEnabled boolean (default false)
   - hlsMasterKey/path (internal storage key, NOT public URL)
   - createdAt, updatedAt

B) STORAGE IN BACKBLAZE B2 (Masked)
1) Upload generated HLS files to B2 under:
   /videos/{videoId}/hls/...
2) NEVER return raw B2 URLs to the frontend.
3) For local dev: allow fallback to local filesystem storage if B2 env vars not set, but keep the same API shape.

C) SECURE DELIVERY APIS (No original link exposure)
Implement these backend endpoints:

1) GET /api/video/manifest?videoId=...
   - Requires auth (use existing auth system).
   - Returns the HLS master manifest content OR a signed internal URL to it.
   - IMPORTANT: The manifest must reference segment URLs that go through YOUR API, not B2.
     Example inside playlist should point to:
       /api/video/segment/{videoId}/{variant}/{segmentName}?token=...
   - Must generate short-lived tokens (JWT) for segment access.

2) GET /api/video/segment/:videoId/:variant/:segmentName
   - Requires a valid short-lived JWT token (e.g., 30–60 seconds).
   - Token must be bound to:
     - userId
     - sessionId (generated on manifest request)
     - optionally IP hash (soft check: allow small IP changes, but block big changes)
   - Server fetches segment from B2 (server-side) and streams it to client.
   - Add correct Content-Type headers for segments and playlists.

3) (Optional but recommended) GET /api/video/playlist/:videoId/:variant
   - Same token rules
   - Returns the variant playlist but with URLs rewritten to API segment endpoint.

D) TOKEN + SESSION RULES
1) Use JWT with:
   - videoId, userId, sessionId, exp, type (manifest/segment)
2) Short expiry:
   - manifest token: 2–5 minutes
   - segment token: 30–60 seconds
3) Rotation:
   - Each time a playlist is requested, issue a fresh session token if needed.
4) Do not store secrets client-side. JWT is sent via query param or Authorization header for segment endpoints.

E) RATE LIMIT + ANTI BULK DOWNLOAD
1) Implement per-user + per-session throttles:
   - Max concurrent segment requests: 3
   - Max segment requests per 10 seconds: configurable (e.g., 20)
2) Detect suspicious behavior:
   - Too fast segment requests
   - Excessive parallel requests
   - Frequent out-of-order segment requests
3) On violation:
   - increment breach counter: breachCount (per user per video)
   - return a structured JSON error:
     { error: "SECURITY_BREACH", breach: "2/6", blockSecondsRemaining: ... }

F) VIOLATION LIMIT + AUTO BLOCK (10 MINUTES)
1) Add settings:
   - violationLimit default 6
   - blockDurationSeconds default 600 (10 minutes)
2) Behavior:
   - Each breach increments count and returns “Security breach X/6”
   - On reaching limit (6/6):
     - block video playback for that user+video for 10 minutes
     - any attempt to fetch manifest/playlist/segment returns:
       { error: "VIDEO_BLOCKED", blockSecondsRemaining: 600 }
3) Store these counters in Replit DB for now, in a structure that can later move to Supabase easily:
   Key pattern examples:
   - security:{userId}:{videoId} -> { breachCount, blockedUntil, lastBreachAt, sessionId }
